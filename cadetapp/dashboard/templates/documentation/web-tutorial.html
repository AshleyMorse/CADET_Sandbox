User Manual Documentation

Johns Hopkins EP Program uses CADET to automatically parse and categorize every semesters' student feedback for each of the class sections in the program. This method is relatively new and lacks any documentation or official tutorial for those that assemble the feedback for review. This specification is for an official web-based tutorial in how to use the tool.

Problem Description

The new CADET software does not have any official documentation that gives a step by step use of the program itself or answers questions a user may have. Every good software program needs to be accompanied by a well written set of user guides that contain all of the aforementioned details. A simple and efficient way to make the user guide accessible to all is a web-based deployment. Thus, DocUtils [DocUtils] was chosen to provide an easy to use script to convert all RST [RST] files containing the developer documentation into Extensible Hypertext Markup Language (XHTML) [XHTML], a web formatting language, for this purpose and shall be hosted on the same site that the UI team uses for their deployment.

Requirements

Functional Requirements - File extension detection to ensure it is a RST document - Parses above file to display in a web page format in XHTML

Quality Requirements - Instructions to build and deploy developer documentation should work across multiple platforms i.e- firefox, chrome, etc.

  • Coding style should be clean, neat, and easy to use

Use Cases

  • Primary Actors: User, localhost
  • Goal: Convert RST files into XHTML and host them on a web server
  • Goal: Instructions to deploy user manual in a web based platform
  • Precondition: - User follows setup guide for DocUtils [DocUtils] - User sets up environment for script - The CADET program has been hosted to an existing web server - An RST style document with correct formatting is part of the repository [RST]

After the user has finished writing their developer documentation, they shall run the 'rst2html.py' script. If there is an error encountered during the conversion process, there is an error output to the terminal with details of the error. If there are errors, the user shall have to rework the RST text file to address these errors and rerun the script. Once all errors are removed, the script will successfully generate all XHTML documents in the directory.

  • Postcondition: - Correctly formatted XHTML files have been generated - XHTML files have been moved into directory for web hosting - A web page that displays the current semester, along with previous semesters implementation once deployed

Proposed Change

This change will create an automatic process for converting the user guide and for ease of access. It will have a menu to navigate through each semester and show the added modules and other functionality along with how to use it.

Alternatives

Alternativs Options: - Text document - Printed document

The web based application for the developer achieves ease of access across multiple platforms allowing anyone to use it with internet access. Versus the alternative options, the web based application will be simpler than using a text editor.

Testing

Testing will consist of two parts. Part one will involve the platform, i.e - the internet browser, that the instructionswork to deploy to while part two involves checking the format of the submitted file to ensure it is a properly formatted RST specific text file.

The test cases would be different browser platforms for the instructions but for simplicity it will be limited to Firefox and Chrome and different text file extensions will be submitted to ensure correct error handling occurs.

Documentation

The web-based documentation is the final product but the script repository will include a readme text file that has all necessary setup steps and a step-by-step process of how to run the tool
